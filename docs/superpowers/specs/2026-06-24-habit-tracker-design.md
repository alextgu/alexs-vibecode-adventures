# Habit Tracker — Design

**Date:** 2026-06-24
**Status:** Approved, ready for implementation plan

A personal daily habit tracker whose visual design is generated by an LLM from the user's diary entries. The user writes diary entries on the calendar (one per day, button-driven); each submission is fed to a model along with a stored "who I am" prompt, and the model rolls a new frontend theme. The dashboard is functionally complete with or without theming — generated themes are the final layer on top of a working app.

## Scope

In:
- Single-user dashboard at `/dashboard` (Clerk-gated)
- Calendar month grid showing per-day score + habit completion density + diary "medal" markers
- Habit definitions with cadences (daily / every-N-days / weekly / monthly)
- Habit completion tracking per day
- Long-term goals with optional target dates
- Diary entries (one per date, free-form text, user submits when ready)
- "Who I am" identity prompt (user-editable, persisted per user)
- AI-generated themes: model is selected per submission; output is a structured token + layout-choice payload that drives the rendered UI
- One hand-coded **default theme** as the baseline / fallback before any diary submission

Out (for v1):
- Public profile / sharing — `public_daily_logs` view is removed
- Calories, protein, and the standalone `journal_submitted` flag (journal becomes a normal habit with `every_n_days = 2`)
- Optimistic UI
- Notifications, reminders, exports
- Multi-user, teams, social features
- Free-form JSX/HTML generation by the model — generation is bounded to a token + layout-choice schema (never raw markup)
- Diary entry length thresholds / auto-submit — the user decides when to submit
- Public-facing themes gallery / sharing of generated themes

## Build sequence

This is the user's explicit constraint:

1. **Phase 1 — Backend & functional UI.** Schema migration, server actions, routes, and a fully working dashboard rendered with the hand-coded **default theme**. No AI involvement. App must be usable end-to-end (set scores, check habits, manage goals, write diary entries).
2. **Phase 2 — Theme generation pipeline.** Diary submission triggers a model call through AI Gateway; output is validated against the theme schema; new theme version persists; dashboard re-renders against the new active theme.

Each phase ends in a fully shippable app. Phase 2 is purely additive to Phase 1 — no Phase 1 surface is rewritten by Phase 2.

## Data model

### `daily_logs` (existing, slimmed)
Drop `calories`, `protein_g`, `journal_submitted`, `completed_habits`. Keep:
- `id uuid pk`
- `user_id text`
- `date date`
- `day_score int check (between 1 and 10)`
- `created_at`, `updated_at`
- Unique `(user_id, date)`

### `habits` (new)
- `id uuid pk`
- `user_id text`
- `name text`
- `cadence_type text check (in 'daily','every_n_days','weekly','monthly')`
- `cadence_interval int default 1` — used by `every_n_days` (e.g. 2 = every other day); ignored otherwise
- `emoji text nullable`
- `color text nullable` — hex string; themes may or may not honor it
- `archived_at timestamptz nullable` — soft-delete; archived habits stop appearing in current views but their historical completions still render
- `created_at timestamptz`

### `habit_completions` (new)
- `id uuid pk`
- `user_id text`
- `habit_id uuid references habits(id) on delete cascade`
- `date date`
- `created_at timestamptz`
- Unique `(habit_id, date)` — one completion per habit per day

### `goals` (new)
- `id uuid pk`
- `user_id text`
- `title text`
- `target_date date nullable`
- `completed_at timestamptz nullable`
- `created_at timestamptz`

### `diary_entries` (new)
- `id uuid pk`
- `user_id text`
- `date date` — the date the entry is "for"
- `content text` — markdown / plain text
- `created_at timestamptz`
- `updated_at timestamptz`
- Unique `(user_id, date)` — one diary per user per day; editing existing entry is an upsert

### `user_profiles` (new)
- `user_id text pk` — Clerk user_id; one row per user
- `identity_prompt text not null default ''` — the "who I am" preamble prepended to every diary submission
- `updated_at timestamptz`

### `theme_versions` (new)
- `id uuid pk`
- `user_id text`
- `source_diary_id uuid nullable references diary_entries(id) on delete set null`
- `model text not null` — model identifier sent to AI Gateway, e.g. `anthropic/claude-opus-4-7`
- `tokens jsonb not null` — colors, fonts, glyphs, copy strings (see Theme contract below)
- `layout_choices jsonb not null` — picks from the curated layout allowlist
- `created_at timestamptz`

Active theme for a user = the most recent `theme_versions` row by `created_at`. If none, the default hand-coded theme renders.

### RLS
All six tables (existing + new) follow the existing pattern:
- `auth.jwt() ->> 'sub' = user_id` for select / insert / update / delete
- `revoke all from anon, authenticated`
- `grant select, insert, update, delete to authenticated`

The `public_daily_logs` view and any `grant select ... to anon` are dropped.

## Server actions

`src/lib/daily-log.ts` is replaced by topic-split files under `src/lib/`:

- `src/lib/days.ts`
  - `setDayScore(date: string, score: number)`
- `src/lib/habits.ts`

  - `toggleHabitCompletion(habitId: string, date: string)`
- `src/lib/goals.ts`
  - `createGoal(input)` / `updateGoal(id, input)` / `completeGoal(id)` / `deleteGoal(id)`
- `src/lib/diary.ts`
  - `saveDiaryEntry(date: string, content: string)` — upsert; does NOT trigger theme generation on its own
  - `submitDiaryForRegen(date: string, model: string)` — upsert latest content + invoke Phase 2 pipeline (see below)
  - `getDiaryEntry(date: string)`
- `src/lib/profile.ts`
  - `setIdentityPrompt(text: string)` — upsert into `user_profiles`
  - `getIdentityPrompt()`
- `src/lib/themes.ts`
  - `getActiveTheme(): Promise<ResolvedTheme>` — returns latest `theme_versions` row, or the default theme constant
- `src/lib/month-data.ts`
  - `getMonthData(yearMonth: string): Promise<MonthData>`:
    ```ts
    type MonthData = {
      daysScored: Record<string, number>;           // ISO date -> 1..10
      completions: Record<string, string[]>;        // habitId -> ISO dates this month
      habits: Habit[];
      goals: Goal[];
      diaryDates: string[];                         // ISO dates with a diary entry this month
    }
    ```

All actions return the existing shape: `{ ok: true, ... } | { ok: false, error: string }`.

Streak and "overdue" computation happens in TypeScript on the server using the returned data; not in SQL.

## Theme contract

The theme is **data**, not React. The component tree is fixed; themes vary by tokens and a small set of layout choices.

```ts
// src/themes/types.ts
type ThemeTokens = {
  colors: {
    bg: string;             // hex
    fg: string;
    muted: string;
    accent: string;
    accent2: string;
    success: string;
    warning: string;
  };
  fonts: {
    body: string;           // CSS font-family value
    display: string;
    mono: string;
  };
  radius: 'none' | 'sm' | 'md' | 'lg' | 'full';
  border: 'none' | 'thin' | 'thick' | 'double' | 'dashed';
  glyphs: {
    checked: string;        // e.g. "✓" or "★" or "[X]"
    unchecked: string;
    today: string;
    medal: string;          // marker for days with a diary entry
    streakFlame: string;
  };
  copy: {
    appTitle: string;
    todayLabel: string;
    streakLabel: string;    // e.g. "{n}-day streak" - placeholder substitution
    overdueLabel: string;
    addHabitCta: string;
    addGoalCta: string;
    writeDiaryCta: string;
    saveCta: string;
    cancelCta: string;
    emptyHabitsState: string;
    emptyGoalsState: string;
  };
};

type LayoutChoices = {
  calendar: 'grid' | 'heatmap' | 'spiral';
  habitsList: 'list' | 'cards' | 'stack';
  goalsList: 'list' | 'cards';
  dayCellEmphasis: 'score-shade' | 'glyph-pile' | 'sparkline';
  shellHeader: 'centered' | 'banner' | 'sidebar';
};

type ResolvedTheme = {
  source: 'default' | 'generated';
  versionId: string | null;
  tokens: ThemeTokens;
  layout: LayoutChoices;
};
```

Layout-choice values are a **closed allowlist**. The model is constrained to picking from these specific strings; any unknown value falls back to the default for that field at validation time.

### Default theme

`src/themes/default.ts` exports a `ResolvedTheme` with `source: 'default'`. Plain, neutral, monochrome. Used when no `theme_versions` row exists for the user.

### Component tree

Components live in `src/components/` (NOT inside a theme folder). They read tokens via CSS variables (set on `<html>` from `tokens.colors` and `tokens.fonts`) and branch on `LayoutChoices` for layout primitives.

- `<Shell>` — chooses header style by `layout.shellHeader`
- `<Calendar>` — chooses grid vs. heatmap vs. spiral by `layout.calendar`
- `<DayCell>` — chooses emphasis by `layout.dayCellEmphasis`; renders medal glyph when `date` is in `monthData.diaryDates`
- `<HabitsList>`, `<GoalsList>` — choose style by their respective layout fields
- `<DayLogModal>`, `<HabitForm>`, `<GoalForm>` — visual only; no layout branching needed

## Theme generation pipeline (Phase 2)

Triggered by `submitDiaryForRegen(date, model)`:

1. Upsert diary content for `date`.
2. Load `user_profiles.identity_prompt`.
3. Call AI Gateway with the selected `model`:
   - System prompt: instructs the model to return a single JSON object matching the theme schema, with the layout-choice enum constraints stated inline.
   - User payload: `{ identity_prompt, diary_content }`.
4. Validate the response with a Zod (or hand-written) schema against `ThemeTokens` + `LayoutChoices`. Any unknown enum value falls back to the default value for that field.
5. Insert a new row into `theme_versions` with the validated payload + source_diary_id + model.
6. `router.refresh()` on the client; next render reads the new active theme.

Errors during generation (timeout, validation failure, gateway error) are surfaced inline in the diary submit UI; the diary content is still saved. The current active theme stays in place — no automatic fallback to default.

## Routes & UX

### Routes
- `/` — sign-out: Clerk sign-in card. Signed-in: redirect to `/dashboard`.
- `/dashboard` — the app. Optional `?ym=YYYY-MM` query for the month being viewed.
- `/dashboard/diary/[date]` — full-page diary editor for a given date.
- `/settings` — edit the "who I am" identity prompt and manage habits/goals administration (single page, two sections).
- `middleware.ts` — Clerk auth gate.

### Dashboard layout
`dashboard/page.tsx` is a Server Component:
1. Resolves active theme via `getActiveTheme()`; sets CSS variables on `<html>` from `tokens.colors` + `tokens.fonts`.
2. Calls `getMonthData(ym)`.
3. Renders `<Shell>` wrapping `<Calendar>`, `<HabitsList>`, `<GoalsList>`, threading the resolved `LayoutChoices` down.

### Interactions
- **Click a day cell** → opens `DayLogModal` for that date: `day_score` 1–10 input + checkbox per habit (filtered to habits not archived before that date) + link "write today's diary" / "edit diary" (jumps to `/dashboard/diary/[date]`).
- **Click the medal glyph on a day cell** → jumps to `/dashboard/diary/[date]` (read/edit).
- **Click a habit row** → opens `HabitForm` in edit mode.
- **Click "new habit" CTA** → opens `HabitForm` in create mode.
- **Click goal checkbox** → calls `completeGoal`.
- **Click "new goal" CTA** → opens `GoalForm`.
- **Month nav** → updates `?ym=` query; server refetches.

### Diary editor (`/dashboard/diary/[date]`)
- Textarea for `content` (autosaves to `saveDiaryEntry` on blur / debounce — does NOT regenerate).
- Model picker (dropdown of allowed AI Gateway models).
- "Submit & regenerate theme" button → calls `submitDiaryForRegen`.
- Inline status: "Generating…" / error message / success.

### Settings (`/settings`)
- Identity prompt textarea (`setIdentityPrompt`).
- Habits administration list (create, edit, archive — same forms used in modals).
- Goals administration list.

### Server / client split
- All `page.tsx` files + `Shell` + list containers: server components.
- Modals, forms, the diary editor, the model picker: client components. They invoke server actions and call `router.refresh()` on success.

## Error handling

- Server actions return `{ ok, error? }`; modals and forms surface `error` as inline text. No toast system.
- Theme-generation errors do not roll back the diary upsert.
- Validation against the theme schema is the final gate on what reaches `theme_versions`. Defaults fill missing fields silently.
- RLS is the final gate on data writes.

## Testing

V1: manual verification only. Manually verify Phase 1 dashboard works against the default theme before starting Phase 2. For Phase 2, verify with one or two real model submissions; add structural unit tests around the theme schema validator if validation logic gets non-trivial.
