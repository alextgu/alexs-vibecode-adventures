# Hard 75 — Design

**Date:** 2026-07-01
**Status:** Approved (user waived per-section approval), ready for implementation plan

A daily-checklist challenge system. The user defines a set of custom boolean rules, starts a 75-day attempt, and must check off every rule every day for 75 consecutive days. Missing any rule on any day fails the attempt and forces a restart at Day 1. History of past attempts is preserved.

The framework is generic (a `challenges` table with `target_days`), but the only preset in v1 is Hard 75. Naming is generic so a "Hard 30" can drop in later with no migration.

The Hard 75 UI is deliberately plain: white background, black text, Arial. This spec ships **standalone** — it does not depend on the earlier habit-tracker spec (`2026-06-24-habit-tracker-design.md`) being implemented first. If that tracker spec is later built on top of this, the `user_profiles` table is already in place and `ChallengeBadge` can move onto `/dashboard`; nothing in this spec has to change for that to happen.

## Scope

In:
- Generic `challenges` framework preset for Hard 75 (`target_days = 75`)
- One active attempt per user at a time; unlimited past attempts as history
- Custom boolean rules, defined at attempt start, locked once the attempt starts
- Timezone stored on `user_profiles` and snapshotted onto each attempt
- Lazy fail check: on any authenticated read of the active attempt, if an uncompleted past day exists, mark the attempt failed on that date
- Home page (`/`) becomes the challenge hub: signed-out shows the Clerk sign-in card; signed-in shows the active attempt with today's checklist, or a start-new form, or the most recent past attempt with a restart CTA
- Dedicated `/challenges` route for the past-attempts list; `/challenges/[id]` for a read-only view of any attempt
- Timezone editable at `/settings`
- Plain white / black / Arial styling everywhere Hard 75 renders

**Prerequisites folded in** (the repo has Clerk + Supabase installed but not wired):
- Add `middleware.ts` using Clerk (`clerkMiddleware` from `@clerk/nextjs/server`) — public routes: `/`. All other routes require sign-in.
- Wrap `src/app/layout.tsx` in `<ClerkProvider>` (from `@clerk/nextjs`).
- Replace the CNA scaffold in `src/app/page.tsx` with the challenge hub.
- Swap out the Geist font imports in `layout.tsx` for `font-family: Arial, sans-serif` applied globally via `globals.css`. Remove the CNA gradients / dark-mode classes.
- The existing `src/lib/daily-log.ts` is left in place, unused, and can be deleted independently if the earlier tracker spec is not going to be built.

Out (for v1):
- Numeric or free-text or photo-proof rules (booleans only)
- Editing rules mid-attempt (must abandon + restart)
- Reminders, push notifications, cron jobs
- Sharing / social / public gallery
- Streak visualizations beyond a plain per-day pass/fail grid
- Multiple concurrent attempts

## Data model

New tables added to `supabase/schema.sql`. All follow the existing owner-only RLS pattern (`auth.jwt() ->> 'sub' = user_id` for select/insert/update/delete; `revoke all from anon, authenticated`; `grant select, insert, update, delete to authenticated`).

### `user_profiles` (new — created here, may already exist if the prior tracker spec landed first)

```sql
create table if not exists public.user_profiles (
  user_id    text primary key,
  timezone   text not null default 'UTC',
  updated_at timestamptz not null default now()
);

alter table public.user_profiles
  add column if not exists timezone text not null default 'UTC';
-- IANA identifier, e.g. 'America/Los_Angeles'. Editable in /settings.
```

Both statements are idempotent so this migration is safe whether the prior tracker spec landed first or not. If the prior spec later adds `identity_prompt`, it does so via its own `add column if not exists`.

### `challenges` (new)

```sql
create table public.challenges (
  id             uuid primary key default gen_random_uuid(),
  user_id        text not null,
  title          text not null default 'Hard 75',
  target_days    int  not null default 75 check (target_days > 0),
  start_date     date not null,
  status         text not null check (status in ('active','completed','failed')),
  failed_on_date date,
  completed_at   timestamptz,
  timezone       text not null,
  created_at     timestamptz not null default now()
);

-- At most one active attempt per user
create unique index challenges_one_active_per_user
  on public.challenges (user_id) where status = 'active';
```

- `start_date` is the calendar date of Day 1 in `timezone`.
- `timezone` is snapshotted at start so fail semantics don't shift if the user changes their TZ mid-attempt.
- `failed_on_date` is set iff `status = 'failed'`. It is the first date on which not all rules were checked.
- `completed_at` is set iff `status = 'completed'`.

### `challenge_rules` (new)

```sql
create table public.challenge_rules (
  id           uuid primary key default gen_random_uuid(),
  challenge_id uuid not null references public.challenges(id) on delete cascade,
  user_id      text not null,
  label        text not null,
  position     int  not null,
  created_at   timestamptz not null default now(),
  unique (challenge_id, position)
);
```

Rules are inserted at attempt start and never updated or deleted afterward. The application layer enforces "no edits after start"; the schema does not.

### `challenge_checks` (new)

```sql
create table public.challenge_checks (
  id           uuid primary key default gen_random_uuid(),
  challenge_id uuid not null references public.challenges(id) on delete cascade,
  rule_id      uuid not null references public.challenge_rules(id) on delete cascade,
  user_id      text not null,
  date         date not null,
  created_at   timestamptz not null default now(),
  unique (rule_id, date)
);

create index challenge_checks_challenge_date_idx
  on public.challenge_checks (challenge_id, date);
```

- One row per rule per day when checked. Absence = unchecked.
- `date` is a calendar date interpreted in the attempt's snapshot `timezone`.
- The unique constraint makes `checkRule` idempotent under double-clicks and races.

### Notes on the model

- No `challenge_days` table. A day D "passed" iff `count(distinct rule_id where challenge_id = C and date = D) == (number of rules for C)`. Computed in TypeScript from `getActiveChallenge` / `getChallenge` payloads.
- Streak = length of the currently active attempt (`today - start_date + 1` days). No separate streak counter.

## Server actions

New file: `src/lib/challenges.ts`. All actions return the existing shape: `{ ok: true, ... } | { ok: false, error: string }`.

```ts
// Read
getActiveChallenge(): Promise<ActiveChallenge | null>
getChallenge(id: string): Promise<ChallengeDetail | null>
getChallengeHistory(): Promise<ChallengeSummary[]>

// Write
startChallenge(input: {
  title?: string;          // default 'Hard 75'
  target_days?: number;    // default 75
  rules: string[];         // required, non-empty, order = position
}): { ok: true; id: string } | { ok: false; error: string }

checkRule(challengeId: string, ruleId: string, date: string): Result
uncheckRule(challengeId: string, ruleId: string, date: string): Result
abandonChallenge(id: string): Result
```

Type shapes:

```ts
type ActiveChallenge = {
  id: string;
  title: string;
  target_days: number;
  start_date: string;              // ISO date, interpreted in `timezone`
  timezone: string;
  current_day: number;             // 1..target_days
  today: string;                   // ISO date in `timezone`
  rules: { id: string; label: string; position: number }[];
  today_checks: string[];          // rule ids checked for `today`
  days_passed: number;             // count of past days where all rules were checked
};

type ChallengeSummary = {
  id: string;
  title: string;
  target_days: number;
  start_date: string;
  status: 'active' | 'completed' | 'failed';
  outcome_day: number | null;      // failed_on_date or target_days, in "day N" form
  completed_at: string | null;
  failed_on_date: string | null;
};

type ChallengeDetail = ChallengeSummary & {
  timezone: string;
  rules: { id: string; label: string; position: number }[];
  checks: Record<string, string[]>;  // ISO date -> rule ids checked that date
};
```

### Lazy fail / complete evaluation

`getActiveChallenge` (and any read of the active attempt) runs this pass before returning:

1. Load the active attempt for the user; if none, return `null`.
2. Compute `today` = current calendar date in `challenge.timezone`.
3. Load rules for the attempt (`R = rules`).
4. Load all checks for the attempt where `date < today`, grouped by date.
5. Let `last_required_date = start_date + target_days - 1`.
6. Walk dates from `start_date` up to `min(today, last_required_date) - 1`:
   - If any date `D` has `checks[D].length < R.length`, set `status = 'failed'`, `failed_on_date = D`, persist with `WHERE status = 'active'` guard, and return `null`.
7. Completion check (in this order):
   - If `today > last_required_date`: all `target_days` days are behind us. Any incomplete would have been caught at step 6; if we got here, mark `status = 'completed'`, `completed_at = now()`, persist with the guard, return `null`.
   - If `today == last_required_date` AND `checks[today].length == R.length`: last day is finished today; same treatment.
8. Otherwise return the `ActiveChallenge` payload.

The `WHERE status = 'active'` guard on the persist makes the operation safe under concurrent requests: whichever write lands first wins; the other is a no-op.

### `startChallenge` details

1. Validate `rules.length >= 1`, all labels non-empty after trim.
2. Load user's `timezone` from `user_profiles`. If missing row, insert with defaults.
3. `start_date` = today's calendar date in that timezone.
4. Insert `challenges` row with `status = 'active'`. The partial unique index rejects a second active row per user; on `unique_violation`, return `{ ok: false, error: 'You already have an active challenge.' }`.
5. Insert `challenge_rules` in a single batch, positions `0..n-1` matching the input order.
6. Return `{ ok: true, id }`.

### `checkRule` / `uncheckRule` details

- Both require that the challenge is `status = 'active'`. Reject with `error: 'Challenge is not active.'` otherwise.
- `checkRule`: insert `(challenge_id, rule_id, user_id, date)`. On `unique_violation`, treat as success (idempotent).
- `uncheckRule`: delete where `(rule_id, date)`. Always `{ ok: true }` even if no row existed.
- `date` must equal `today` in the attempt's timezone. Checking rules for a past or future date is rejected. This preserves the "midnight local = fail" rule — you can't retroactively rescue a missed day.

### `abandonChallenge` details

Sets `status = 'failed'`, `failed_on_date = today` (in the attempt's TZ). Manual give-up path.

## Routes & UX

Plain white background, black text, `font-family: Arial, sans-serif` everywhere in this feature. Nothing here consumes theme tokens. If the earlier tracker spec is later built, its themed surfaces stay themed; the Hard 75 surfaces stay plain.

The app currently has only `/` (Clerk sign-in). This spec replaces `/` post-signin with the challenge hub, and adds `/challenges`, `/challenges/[id]`, and `/settings`.

### Routes

- `/` — signed-out: Clerk sign-in card (unchanged). Signed-in: **the challenge hub**. States:
  - **No attempts yet**: renders the start-new form (title, target_days, rules editor).
  - **Active attempt**: renders `{title} · Day {n} / {target_days}`, today's checklist with checkboxes, an "Abandon" button, and a small link to `/challenges` for history.
  - **No active but past attempts exist**: renders the most recent past attempt's outcome (`Completed` trophy or `Failed on Day N`) plus a start-new form below.
- `/challenges` — full past-attempts list. Each row: title, start_date, status, outcome (`Day N failed` / `Completed`). Click → `/challenges/[id]`.
- `/challenges/[id]` — read-only detail for a past/completed attempt. Snapshotted rules, per-day pass/fail grid across `target_days`, outcome.
- `/settings` — timezone select (IANA list) that writes to `user_profiles.timezone`. Changing timezone does NOT affect the currently active attempt (it uses its snapshot); it applies to future attempts.

### Start-new form

Rendered on `/` when there is no active attempt.
- Title input, default `Hard 75`.
- Target days input, default `75`.
- Rules editor: dynamic list of text inputs, add row / remove row / drag-to-reorder. Minimum 1 rule. Empty rows trimmed on submit.
- Confirm & start button → `startChallenge`. Errors render inline.

### Active-attempt view (on `/`)

- Header line: `{title} · Day {n} / {target_days}`.
- Checklist: each rule is a checkbox row (`<label>` wrapping `<input type="checkbox">`). Checkbox toggles call `checkRule` / `uncheckRule` and `router.refresh()`.
- When all rules are checked: small `Day {n} complete ✓` line under the list.
- Abandon button: opens a plain confirm dialog; on confirm, calls `abandonChallenge` and refreshes.

### Completed / failed post-state (on `/`)

- Completed: `{title} · {target_days}/{target_days} ✓ completed on {date}` — permanent trophy line until the user starts a new attempt (they don't have to).
- Failed: `{title} · failed on Day {n}` — same treatment.
- Both are followed by the start-new form so restarting is immediate.

## Timezone

- Stored on `user_profiles.timezone` (IANA string, default `'UTC'`).
- Editable in `/settings`.
- Snapshotted onto each `challenges` row at start. Attempt evaluation always uses the snapshot, not the current `user_profiles.timezone`.

## Server / client split

- `/`, `/challenges`, `/challenges/[id]`, `/settings` — server components; call `getActiveChallenge` / `getChallenge` / `getChallengeHistory` on the server.
- Checklist row, start-new form, abandon button, timezone select — client components. They call server actions and `router.refresh()` on success.

## Error handling

- Server actions return `{ ok, error? }`; forms and the badge surface `error` as inline text. No toast system.
- Double-start race is absorbed by the partial unique index; the losing request gets a friendly error.
- Double-check race is absorbed by the checks unique constraint; both writes succeed idempotently.
- Concurrent lazy-fail writes are guarded by `WHERE status = 'active'`; whichever lands first wins.
- RLS is the final gate on data writes.
- Time skew between client and server has no effect: `today` is always computed on the server from the attempt's snapshot timezone.

## Edge cases

- **Timezone change mid-attempt:** the attempt keeps its snapshot TZ. Newly started attempts pick up the new TZ.
- **DST transitions:** dates are calendar dates in the snapshot TZ. A 23- or 25-hour day still ends at local midnight; nothing changes about the check window's *date* boundary.
- **User doesn't visit for several days:** the next visit's lazy-fail scan walks past days in order and fails on the first incomplete one, setting `failed_on_date` to that specific day (not "today").
- **User completes the last day:** checking the final rule on Day `target_days` triggers the completion path on the next read (immediately, since check + refresh runs `getActiveChallenge`). There is no explicit "finish" click.
- **Zero-rule attempt:** rejected at `startChallenge`.
- **User archives / abandons and immediately restarts same day:** allowed. The new attempt's `start_date` is today; the abandoned attempt shows `failed_on_date = today` in history.

## Testing

Manual verification only in v1, matching the existing project convention:
- Start an attempt with 3 rules; check all today; reload — badge shows Day 1 complete.
- Change device clock forward one day without checking — next reload marks attempt failed on the previous day.
- Start a new attempt with `target_days = 2`; complete both days; on day 3 reload — status flips to completed.
- Start two attempts back-to-back in the same request cycle (simulate race) — one succeeds, one returns the "already active" error.
- Verify `/challenges/*` and the badge render in white/black/Arial regardless of the active theme.
